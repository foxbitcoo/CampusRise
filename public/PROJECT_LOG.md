## 2026-05-06 第二次整理

### 本次判断

- 原始提示词本质上是“人物侧脸剪影 + 校园叙事填充 + 电影海报/梦幻水彩质感”
- 原始写法偏口语化，更适合临时调用，不适合直接做技能输入层
- 当前最大问题不是画风不够，而是输入结构不够清楚，导致稳定性和差异化都有限

### 本次决定

- 提示词改为“槽位式输入”
- 槽位分为必填和选填两类
- 生成前优先提醒用户补全可填信息，以提高成图成功率
- 产品方向先收敛成两个版本：
  - 应届毕业版
  - 校友回望版

### 校友回望版的意义

- 不再只服务“马上毕业的人”
- 可覆盖毕业多年后的母校回望、身份归属、校史连接、校友精神映射
- “校史传承”和“校友精神”先合并进这个版本中迭代

### 交互层新增决定

- 不拆成两个独立 skill
- 保留一个统一 skill 入口
- skill 内部自动分流到“应届毕业版”或“校友回望版”
- 只有在判断不清楚时，才补问用户一句

### 工作流层新增决定

- skill 入口不是直接生成，而是先进入工作流
- 工作流先做主题判断，再补槽位，再确认，再生成
- 需要支持用户中途纠正，并返回上一步重新选择
- 项目名称开始从“毕业照工具名”转向“母校连接 + 正向激励”的方向思考

## 2026-05-06 第三次整理

### 命名最终决定

- 中文名定为：`校望`
- 英文名定为：`CampusRise`

### 命名原因

- 不再使用 `Alma` 这类对国内用户理解门槛较高的词
- 使用更直接、更容易理解的 `Campus`
- 保留 `Rise`，强调毕业后继续向上、继续生长
- 中文“校望”同时兼顾回望母校与展望未来

### 测试集策略新增决定

- 不批量抓取真实素人照片并打学校假标签
- 学校侧使用真实公开学校元数据
- 人像侧改用本地虚构测试人物或合成占位素材
- 测试目标先聚焦工作流、分流逻辑和 prompt 结构验证

## 2026-05-06 第四次整理

### 工作流重构决定

- 不再先让用户填很多槽位
- 统一改成“先看图、再问人”的工作流
- 第一步先从图片判断人物年龄层
- 第二步先从图片识别学校线索
- 只有学校识别不到时，才向用户补问学校名称

### 分流规则更新

- 以 `30 岁` 作为默认分界线
- `30 岁及以下` 优先走应届毕业方向
- `30 岁以上` 优先走校友回望方向
- 如果用户后续说明自己是博士应届等特殊情况，允许回退修正分支

### 图片输入规则更新

- 不再强行要求用户必须同时上传正脸和侧脸
- 用户只上传一张图时，也要尽量从图里还原人物特征
- 如果图片里恰好有侧脸或 3/4 侧脸，就优先拿来强化剪影轮廓

### 学校语言规则更新

- 中国大陆高校：统一用简体中文
- 香港 / 澳门 / 台湾高校：使用当地常用中文写法；没有更明确要求时，优先用繁体中文
- 海外高校：统一用英文
- 用户只说“清华大学”时，默认指北京的清华大学，使用简体中文“清华大学”
- 只有用户明确说“台湾清华大学”“新竹清华大学”或相关表达时，才按台湾高校处理为繁体中文

### 提示词版本归档

- `V1`：原始小红书版
- `V2`：上一轮双分支优化版
- `V3`：当前统一工作流版

## 2026-05-06 第五次整理

### 用户交互话术更新

- 不再对用户暴露“校友回望版 / 应届毕业版”这类内部版本名
- 改成用户能理解的话术：
  - “我判断您更像是刚毕业或即将毕业的同学”
  - “我判断您更像是已经毕业比较久的同学”

### 学校确认规则更新

- 学校名称必须和用户确认
- 即使模型从图片里识别到学校，也要请用户确认
- 学校确认后，再进入学校元素、校训、校友参考检索

### 学校检索规则新增

- 确认学校后，要检索学校代表性视觉元素
- 要检索校训，并从校训中提取精神关键词
- 要检索真实、知名、正向的校友案例
- 校友不要求用户输入，由模型主动查找并在最终返回里说明
- 不确定的信息不要编造

### 槽位文案更新

- `个人记忆关键词` 改为 `个人特质 / 希望呈现的气质`
- 增加示例：勇敢、自由、坚定、温柔、理性、热烈、独立、探索、创造
- `校园专属经历` 改为 `校园经历 / 社团经历`
- 增加示例：学生会、辩论队、合唱团、实验室项目、支教、创业比赛、图书馆自习、操场夜跑

### 主题方向更新

- 所有生成都要保持积极、昂扬、向上
- 学校元素、校训、校友参考都要服务于用户对母校的自豪感和继续前行的力量感

## 2026-05-06 第六次整理

### 测试流程修正

- 跑测试集时不再 mock 学校名。
- 图片里识别不到学校时，必须像真实 skill 一样向用户确认学校。
- 图片里识别到学校时，也必须向用户确认，用户纠正后再继续。
- 测试结果应优先直接返回在对话流里，不能只写入本地 Markdown。

### 生成质量规则补充

- 应届毕业方向如果用户没有提供毕业年份，默认读取当前时区下的年份。
- 当前环境日期为 2026-05-06，因此默认应届毕业年份应为 2026，不能让图像模型自由生成 2025 等错误年份。
- 校友回望方向如果用户没有提供毕业年份，不编造具体年份，只表达“毕业多年”“回望母校”。
- 人物侧脸必须是第一视觉焦点，不透明度和清晰度要高于背景校园元素。
- 双重曝光只能用于人物轮廓内部叙事和边缘过渡，不能把五官、头发、眼镜、脸型压得过淡或变糊。

## 2026-05-07 第七次整理

### 生成交互更新

- 学校确认后，系统只给用户一次补充专业、个人特质、校园经历、签名名等可选信息的机会。
- 用户没有补充时，不再等待“是否继续生成”的二次确认，直接生成图片或最终提示词。
- 生成后的返回说明承担解释作用：说明使用了哪些学校元素、校训精神、校友线索和色调。

### 色调规则更新

- 学校检索中新增 `学校代表色 / 校徽色 / 校园代表色`。
- 画面整体色调优先参考校徽、官方校色或学校最有代表性的校园颜色，以增强用户对母校的亲近感。
- 不确定是否为官方校色时，只作为视觉灵感使用，不编造“官方校色”。

### Bad Case 规则补充

- 发现生成图可能把上传女性照片生成成男性侧脸，需将 `性别呈现 / 年龄阶段 / 发型 / 五官 / 脸型` 作为人物身份一致性硬约束。
- 发现生成图可能额外生成一个正面全身毕业生主体，偏离“人像侧脸作为背景主视觉”的原目标。
- 发现生成图可能出现多手、多指、断手、融合手指或不合理握持证书的问题，需将肢体稳定性作为硬约束。
- 新规则：画面只允许一个主要人物表达，即大侧脸肖像 / 剪影；毕业仪式可用小尺度人群、远景剪影或象征性道具表达，不额外生成第二个真人主体。

## 2026-05-08 第八次整理

### 版本命名更新

- `V1` 不再作为后续测试和对比候选。
- 当前候选版本收敛为：
  - `V1.5`：校园锚定版，保留一定海报感，但需要控制浮夸程度。
  - `V3`：统一工作流版，更偏真实人物和干净底色。
  - `V4`：原 `Latest` 重命名版本，主推更写真、更干净、更商业可用的方向。

### 人物还原主旨更新

- 多张同一人物照片输入时，应综合正脸、侧脸、3/4 侧脸、全身和环境图建立人物身份模型。
- 多图不是为了固定复制某一张照片的姿势，而是为了更稳定地还原人物原本长相、脸型、发型、气质、年龄阶段和表情可能性。
- 生成结果可以适度美化，并增加面部细节、光影层次和表情表现。
- 关键判断标准是：熟人一眼能认出原主人，同时不会觉得太假、太网红脸或太浮夸。

## 2026-05-13 V4.1 发布前收敛

### 发布场景收敛

- 发布版先聚焦“小红书应届毕业纪念图”。
- 校友回望不再作为默认主流程，只在用户明确要求校友、返校、毕业多年回望时触发。
- 用户如果已明确学校名称，不再二次确认学校，直接进入学校研究和生成。

### 交互脚手架更新

- 学校未知时，追问信息分成两层：
  - `B. 学校信息`：学校名称或对识别学校的确认/更正，这是必要信息。
  - 可选项：姓名、专业 / 学院、校园经历 / 社团经历、想呈现的个人特质、毕业时间、签名名。
- 可选项新增 `姓名`，支持中文名或英文名。
- `毕业时间` 支持年份、月份、季节或具体日期；未声明时，应届毕业方向默认当前年份。
- 如果用户一开始只给了学校，生成后再提示补充姓名、专业、经历、特质、年份等信息可以继续优化，不在生成前增加摩擦。

### 学校高保真规则更新

- 学校建筑、校徽、校门、题字、关键符号等可核对元素不再只靠文字描述。
- V4.1 允许在学校研究阶段多花时间做网页搜索，并下载 1-3 张官方或高可信参考图。
- 有参考图时再做可见学校元素；没有参考图时应缩小、线稿化、剪影化、氛围化，或省略。

### 版本文件

- 新增 `public/prompt_versions/V4.1.md` 作为发布候选。
- 保留 `public/prompt_versions/V4.md` 作为上一版回溯。
- 保留 `public/prompt_versions/LATEST.md` 作为 1.0 历史发布回溯。

### 发布维护能力

- README 增加自动更新说明：GitHub 仓库是最新版本来源；clone 用户可 `git pull origin main`；Codex 安装用户可重新执行安装命令覆盖本地副本。
- Skill 内增加 Auto Update 规则：用户要求更新时，优先从 `foxbitcoo/CampusRise` 拉取并同步本地安装；支持 `SessionStart` 的环境可在新会话开始时自动同步。
- Bad Case 反馈升级为 `report to issue` 工作流：有 issue 编号或链接时优先评论，没有时新建 issue；没有 GitHub 写权限时输出可手动提交的 issue/comment 草稿。
